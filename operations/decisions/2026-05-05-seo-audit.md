### 1. [ALL CATEGORY + SKILL PAGES] Double brand suffix in `<title>` — `… | SKILON | SKILON`
   - **Actual** (verified live):
     - `/skills/category/indian-business` → `Claude Skills for Indian Business — GST, Tally, Invoicing | SKILON | SKILON` (77 chars, will get truncated by Google)
     - `/skills/category/iot` → `… | SKILON | SKILON` (75 chars)
     - `/skills/category/developer-tools` → 72 chars
     - `/skills/category/trading-finance` → 75 chars
     - `/skills/category/ai-llm` → 77 chars
     - `/skills/gst-invoice-generator` → `GST Invoice Generator — SKILON | SKILON` (41)
     - `/skills/stock-screener-ai` → `AI Stock Screener — SKILON | SKILON` (37)
   - **Cause**: root layout sets `title.template: "%s | SKILON"` AND each page's metadata already appends ` | SKILON` (categories) or ` — SKILON` (skill detail).
   - **Fix (1 line each)**:
     - `products/01-claude-reseller/app/src/app/layout.tsx:27` — change `template: "%s | SKILON"` to `template: "%s"` (or, better, keep template and strip the manual suffixes from per-page metadata).
     - Alternatively, in `src/lib/categories.ts` (where `cat.titleTag` is defined), drop the trailing ` | SKILON` from each entry — and in `app/skills/[slug]/page.tsx:~63` drop the ` — SKILON` suffix from the per-skill `title:` value.
   - **Pick one approach and do it consistently.** Recommended: keep `template: "%s | SKILON"` and remove the suffix from category/skill metadata.

### 2. [HOMEPAGE `/`] No canonical URL, no `og:url`, no JSON-LD schema
   - **Actual**: zero `<link rel="canonical">`, zero `og:url`, zero `<script type="application/ld+json">` blocks in served HTML.
   - **Expected**: canonical = `https://addon90days.vercel.app/` (and after cutover, custom domain), `og:url` matching, plus an `Organization` + `WebSite` JSON-LD with `SearchAction` pointing at `/skills?q={query}`.
   - **Fix**: add to `products/01-claude-reseller/app/src/app/layout.tsx` metadata block (line 24-55):
     ```ts
     metadataBase: new URL(process.env["NEXT_PUBLIC_APP_URL"] ?? "https://addon90days.vercel.app"),
     alternates: { canonical: "/" },
     openGraph: { ...existing, url: process.env["NEXT_PUBLIC_APP_URL"] ?? "https://addon90days.vercel.app" }
     ```
   - Then drop a JSON-LD `<script>` directly inside the homepage component (`src/app/page.tsx`, before the hero) with Organization + WebSite types. This is the highest-leverage 2-minute fix on this audit.

### 3. [`/skills`] Same: no canonical, no `og:url`, no JSON-LD — and it's `"use client"`

   - **Cause**: every one of these is `"use client"` (verified line 1 of all 5 page.tsx files), which prevents `export const metadata`.

### 5. [HOMEPAGE + `/skills` + ALL 5 PRODUCT PAGES + `/sign-in` + `/sign-up`] `BAILOUT_TO_CLIENT_SIDE_RENDERING`
   - **Verified live**: served HTML contains `<template data-dgst="BAILOUT_TO_CLIENT_SIDE_RENDERING">` and the `<main>` is empty until JS hydrates. The hero, categories, featured skills — none are in initial HTML.
   - **Impact**: Googlebot can render JS but with budget penalty; non-Google crawlers (Bing, GPTBot, ClaudeBot, social previewers) see almost no body content. This is the single biggest SEO risk before launch.
   - **Cause**: a parent component is using a client-only hook (likely `useSearchParams()` in a Suspense boundary, or the ConditionalSidebar, or the chat widget). The category pages and skill detail pages do NOT bail out → so the issue is layout-level for marketing routes.
   - **Fix**: identify the offender (start by removing the chat widget client wrapper from layout and re-deploying a preview; then ConditionalSidebar). Whichever client tree is wrapping the marketing surface above an SSR component needs to be lowered. Acceptance test: `curl -s https://… | grep -c BAILOUT_TO_CLIENT_SIDE_RENDERING` returns `0`.

### 6. [`/skills/gst-invoice-generator` and all skill detail pages] Meta description = tagline only (too short)

## Recommendation

**FIX-N-THEN-SHIP.**

The site has excellent bones (TTFB <500ms, valid JSON-LD where present, clean security headers, proper sitemap/robots/llms.txt, working OG image pipeline) but six SEO-critical issues will materially hurt the SKILON launch:

- Items #1, #4, #5 will produce visibly broken or duplicate SERP entries on launch day.
- Items #2, #3, #6 leave 75% of pages with sub-optimal meta and zero schema.
- All six can be fixed in a single 2-3 hour focused session.

**Sequence**: fix #5 (CSR bailout) first — it's a single-file diff and unlocks everything; then #1 (title template) one-line fix; then #4 (5 product page conversions, ~30 min); then #2, #3, #6 in parallel. Post-fix verify with `curl -s <url> | tr '>' '\n' | grep -i title` for each surface — should be 60 chars or less, no doubled brand suffix, body content present.

After that, the remaining ~13 items in this report are polish you can clear in the week post-launch without affecting SERP.
