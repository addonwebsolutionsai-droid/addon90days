---

**[3/6]**

What it generates:

- ESP-IDF or PlatformIO scaffold (your choice)
- WiFi + MQTT, or BLE, or HTTP — pick one
- NVS-backed credential storage (no hardcoded secrets)
- OTA stub with rollback flag (so you wire it in at day 0, not day 60)
- Telemetry loop with typed sensor read stubs
- MQTT topic schema matching the sensors you specified

The scaffold compiles and runs. You add business logic on top.

---

**[4/6]**

Usage inside Claude Code:

```
/iot-firmware-scaffold
  board: ESP32-S3
  connectivity: MQTT
  sensors: DHT22, soil-moisture
  include-ota: yes
  platform: ESP-IDF
```

Claude reads the SKILL.md context, generates the directory tree,
all boilerplate .c/.h files, the CMakeLists, and a README with pin mapping.

If you spec MQTT, you get topic definitions for telemetry, commands, and status.
If you spec BLE, you get GATT service scaffolding.

---

**[5/6]**

What this skill is NOT:

- It does not write your sensor driver or calibration logic
- It does not replace knowing your hardware
- It does not handle board bring-up on novel silicon
- It is a time-saver for engineers who already know what they are building

The goal is 30 seconds to a compilable skeleton, not 45 minutes of copying
from a prior project and forgetting to update the NVS namespace.
